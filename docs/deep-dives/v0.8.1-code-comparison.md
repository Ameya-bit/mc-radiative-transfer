## 1. What the code comparison proves that the analytic check cannot

The analytic check showed the pipeline reproduces a formula we wrote down ourselves — it
verifies the *plumbing* (geometry, bending, visibility, integration) but depends
on no one else's physics. The code comparison asks a different question: **does our machinery
agree with the independent, exact ray-tracing codes the community trusts?** If it
does, a later disagreement with a real star (the beaming comparison and its real-star anchor) is
much more likely to be *physics* (the beaming systematic) than a bug in our bending.

The benchmark is the NICER code-comparison suite,
[Bogdanov et al. (2019)](../paper/references.md#bogdanov-et-al-2019--paper-ii-l26)
("Paper II", L26), where several independently developed codes (Illinois–Maryland,
Columbia, NASA GSFC, Amsterdam/X-PSI) computed agreed-upon synthetic pulse
profiles and converged to ≲ 0.1%. Their reference profiles — generated by the
Illinois–Maryland (IM) code — are published as supplementary data, so we can
compare directly.

---

## 2. Choosing the one test case our physics can reach

The suite spans rapidly rotating, oblate, energy-resolved stars — most of which
need physics we deliberately defer (Doppler, aberration, oblateness; see
`to_finish.md §7`). The trick is to pick the case where all of that vanishes.
Their **Test SD1a** (Table 1) is exactly that:

| SD1a parameter | value | why it suits us |
|---|---|---|
| ν at infinity | **1 Hz** | surface `v/c ≈ 2.5×10⁻⁴` → Doppler/aberration ≪ 0.1%: pure light bending |
| spot angular radius | **0.01 rad ≈ 0.6°** | effectively a **point spot** — our exact model |
| emission | isotropic Planck | our default isotropic beaming |
| `i`, `θ_s` | 90°, 90° | spot crosses fully behind the star (a hard bending test) |
| `M`, `R` | 1.4 M⊙, 12 km | compactness `u = 2GM/Rc² ≈ 0.3445` |

Two facts make the comparison clean despite our being bolometric while their
output is monochromatic at 1 keV:

- **Redshift** is a constant flux scaling at 1 Hz, so it cannot change the
  *normalized* shape or the pulsed fraction.
- With **isotropic** emission and no Doppler, the only phase dependence is
  geometric (achromatic), so the normalized profile is the *same in every energy
  band* — our bolometric curve and their 1 keV curve must share one shape.

So we normalize both to their peak and compare shapes. The "SD" in SD1a is
"Schwarzschild + Doppler"; at 1 Hz it is the Schwarzschild limit.

## 3. The result — agreement to the Beloborodov-approximation floor

Because `i = θ_s = 90°`, the geometry reduces to `cos ψ(φ) = cos φ`: the spot is
face-on at φ = 0 and directly behind the star at φ = π. With `u ≈ 0.344` the
visibility threshold is `cos ψ ≥ −u/(1−u) = −0.526`, so the spot sets only for
`|φ| > 121.7°` — bending shrinks the eclipse from a flat-space 180° down to
**116.6°**, i.e. we see **≈ 32° around the back** on each side. It is a true
eclipse (`F_min = 0`, `PF = 1`); the test is the **profile shape** and the
**eclipse width**, not the (trivial) pulsed fraction.

![Our SD1a waveform on the IM reference, with the residual peaking at the grazing eclipse edge](../../data/pulse_profile_code_comparison.png)

Overlaying our `point_spot_flux` on the IM reference:

- **Visible fraction: 0.6797 (ours) vs 0.6797 (IM)** — the eclipse ingress/egress
  land on the same phase to the grid.
- **Worst disagreement: 0.81%** of peak, at phase ≈ 0.33 — sitting exactly on the
  eclipse edge, where the emission grazes the limb.
- **RMS over the rotation: 0.18%**; the bulk of the profile agrees to 0.01–0.05%.

That residual shape is the signature of our **one** approximation relative to the
IM code: the [Beloborodov (2002)](../paper/references.md#beloborodov-2002) *linear*
bending map is exact to ~1% for `u ≲ 0.5`, and its error is largest at grazing
emission angles — precisely the eclipse edge. So we reproduce the community
reference to **~1% (max 0.8%, RMS 0.2%)**, limited by the bending approximation
rather than by any bug. That is a legitimate pass of the code-comparison check:
"agrees with the NICER code-comparison codes to ~1%."

## 4. Reproducibility and the reference data

The reference file is third-party AAS supplementary data and is **not committed**
(licensing + provenance are spelled out in
[`../paper/references.md` → Reference data sets](../paper/references.md#reference-data-sets)).
To reproduce: download the L26 supplementary archive, extract it to
`data/l26_reference/`, then run `python scripts/code_comparison.py`. The
`test_sd1a_*` case loads the same file and pins the agreement to < 1.2%; it
**skips cleanly** when the file is absent, so a fresh checkout still passes.

## 5. Status

- **The code-comparison check passes** for test SD1a: max |Δ| = 0.81%, RMS = 0.18%,
  identical visible fraction, against the IM community-reference waveform.
- **Tests: 36/36 pass** (one new code-comparison case; skips without the local reference).
- **No module change:** this is the existing v0.8.0 `point_spot_flux` evaluated at
  a published geometry — which is the strongest possible statement, since the same
  code that passed the analytic check also matches an independent exact code.
- **Honest scope:** SD1a is the *only* suite case fully inside our deferred-physics
  boundary. The finite-spot case **SD1b** (57° cap, same 1 Hz geometry) is the
  natural next extension once a circular-cap tiling is added; the 200/400 Hz cases
  need rotation physics that stays out of scope.
